# Baki Rx Ledger: System Architecture & Development Guidelines

## 1. Project Overview
Baki Rx Ledger is a multi-tenant, local-first SaaS application built for pharmacy owners to manage customer credit (baki) across multiple branch terminals. The system must operate seamlessly offline during internet outages and automatically sync data to the cloud when connectivity is restored, ensuring zero operational downtime at the sales counter.

## 2. Core Technology Stack
* **Frontend Framework:** React Native (Expo/CLI) optimized for Android Studio build targets.
* **Local Storage:** `expo-sqlite` (or WatermelonDB) for offline-first transactional data.
* **Cloud Database:** Supabase (PostgreSQL).
* **Authentication:** Supabase Auth (Online) + SQLite Cryptographic Hashing (Offline Fallback).
* **Network Listener:** `@react-native-community/netinfo` for triggering background sync loops.

## 3. System Architecture (Local-First Hybrid)
The architecture relies on an Append-Only Ledger system to prevent data collisions across multiple offline devices.

* **Read Operations:** The UI strictly reads from the local SQLite database to ensure zero latency. It never awaits a cloud response to render a screen.
* **Write Operations:** Transactions are written locally first with a flag `is_dirty = 1`.
* **Background Sync Engine:** A silent service listens for network connectivity. Upon connection, it pushes all `is_dirty = 1` rows to the cloud, pulls new rows generated by other terminals via timestamps, and resets the local dirty flags upon a 200 OK response.
* **Delta Math:** We do not sync raw "Due Balances" to the cloud. We sync transaction deltas (+100 BDT or -50 BDT). The local device calculates the final balance dynamically.

## 4. UI/UX Design System
The UI is utilitarian, high-contrast, and optimized for fast, one-handed operation under harsh store lighting.

### Brand & Styling Rules
* **Primary Color:** Deep Healthcare Green (`#218868` or similar).
* **Alert/Negative Color:** Crimson Red (`#D92D20`) - exclusively used for due balances and errors.
* **Neutral Backgrounds:** White and light Slate Gray (`#F8FAFC`).
* **Typography:** Inter or Roboto. Financial numbers must be bold and highly legible.
* **Tap Targets:** Buttons and inputs must be massive (min 48px height) for fast typing.
* **Keyboard Behaviors:** Always invoke `keyboardType="numeric"` for mobile numbers and currency inputs.

### Screen Map
* **Terminal Activation (Login):** Includes a branch terminal dropdown selector, mobile number input, 4-digit PIN input, and a green "Network Safe: Offline Mode Ready" indicator.
* **Dashboard (Branch View):** Displays "Total Baki" (Red) and "Today's Collections" (Green) cards. Features large action buttons for "New Baki Entry" and "Add Customer", followed by a list of "Recent Defaulters".
* **New Baki Entry:** A receipt-style form containing Customer Search, Total Bill Amount, Paid Amount, and an auto-calculating red "Due Balance" block. The save button must indicate "(Offline Safe)".
* **Add Customer Modal:** A simple overlay form requesting Customer Name and Phone Number.
* **Customer Profile & Ledger:** Displays the specific customer's Total Due. Includes action buttons for "WhatsApp", "Call", and "Collect Cash". The bottom half contains a chronological Transaction History table (Date, Description, Debit, Credit, Balance).

## 5. Supabase & Multi-Tenant Setup
The cloud database must enforce strict multi-tenant isolation so clerks at Branch A cannot access Branch B's data.

### Database Tables (PostgreSQL)
* `tenants` (id, business_name)
* `stores` (id, tenant_id, store_name, location)
* `customers` (id, store_id, name, phone, updated_at)
* `ledger_entries` (id, store_id, customer_id, total_amount, paid_amount, due_amount, entry_type, created_at)

### Row-Level Security (RLS) Implementation
Every query from the React Native app must be filtered by the `store_id` embedded in the authenticated user's JWT.
Example Policy for `ledger_entries`:
`USING (store_id = (auth.jwt() -> 'user_metadata' ->> 'store_id')::uuid)`

## 6. Future Custom API Integration Scope
The architecture is deliberately decoupled to allow transitioning off Supabase to a custom Virtual Machine backend in the future.

* **API Decoupling:** The local sync engine interfaces only with JSON payloads. The sync service class should be written to easily swap `supabase.from().upsert()` with standard `fetch()` or `axios` POST requests.
* **Backend Tech Choice:** A high-concurrency language like Go or a robust MVC framework like Laravel is recommended for the future API VM to handle heavy JSON delta-sync loads from multiple stores simultaneously.
* **Auth Migration:** The local SQLite session management currently holding Supabase metadata will remain unchanged. The future custom API will simply need to return a standard JWT containing the same `store_id` and `tenant_id` claims upon login.

## 7. Environment Variables Configuration (.env)

To securely manage cloud endpoints and ease the transition to your future API-based VM, the application uses native Expo environment variables.

### Local `.env` File Setup
Create a `.env` file in your root directory. In Expo, all environment variables intended for the client application **must** be prefixed with `EXPO_PUBLIC_`.

```env
# Supabase Configuration (Current)
EXPO_PUBLIC_SUPABASE_URL=[https://your-supabase-project.supabase.co](https://your-supabase-project.supabase.co)
EXPO_PUBLIC_SUPABASE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

# Future API VM Configuration (Pre-configured placeholder)
EXPO_PUBLIC_FUTURE_API_URL=[https://api.bakirxledger.com/v1](https://api.bakirxledger.com/v1)
EXPO_PUBLIC_API_MODE=supabase # Switch to 'custom' when moving to your own VM

# Local SQLite Encryption/Security Salt (Optional)
EXPO_PUBLIC_LOCAL_CRYPT_SALT=your-secure-local-salt-string
